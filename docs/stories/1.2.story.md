# Story 1.2: Subkit and Item Data Architecture

## Status: Done

## Story

**As a** developer,
**I want** a structured, typed data file defining all subkit categories and their associated items,
**so that** all application features can consume a single source of truth for kit content.

## Dependencies

- Requires Story 1.1 (Project Scaffolding) to be complete

## Acceptance Criteria

1. A typed TypeScript data structure defines all 9 subkit categories (name, id, color, size options)
2. All items for each of the 8 predefined subkits are defined (name, id, parent subkit)
3. Custom subkit category is defined and references all items across all other categories
4. Data structure supports item quantity configuration
5. Data structure includes fields for future branded product mapping (productId, price placeholder)
6. All data is importable as a module by any component in the application
7. Slot calculation utilities correctly compute slot state from subkit selections with 100% branch coverage

### Slot Calculations
[Source: docs/architecture/5-state-management.md#slot-calculation-pure-functions]
Full implementation is specified in architecture. These are the most safety-critical functions — 100% branch coverage mandatory.
- `calculateTotalSlots()` — sum of regular(1) + large(2)
- `canFitSize()` — hypothetical replacement check
- `calculateSlotState()` — always returns 6 SlotState[], fills top-to-bottom, handles large blocks
- `isSlotsAtCapacity()` — total >= MAX_SLOTS(6)
- `CATEGORIES` record is used inside `calculateSlotState` to resolve names and colors

### Physical Product Specs
[Source: docs/prd/6-epic-details.md]
- Regular: 6in H x 24in L x 12in W = 1 slot
- Large: 12in H x 24in L x 12in W = 2 slots

### Testing

[Source: docs/architecture/9-testing-requirements.md]
- `slotCalculations.ts` requires 100% branch coverage — most safety-critical code
- Test file: `tests/unit/slotCalculations.test.ts`
- Full test cases are specified in architecture — follow exactly
- Framework: Vitest with `describe`, `it`, `expect`
- IMPORTANT: Overflow test must use real category IDs (e.g., 'power', 'lighting', etc.) so that `calculateSlotState` actually fills slots. Using fake IDs like 'cat0' would cause the category lookup to return undefined and skip all items, making the test pass vacuously.
