# Story 18.1: Fill Your Kit Screen — Shell and Category Sections

## Status: Review

## Story

**As a** user who completed my kit order,
**I want** to see my subkits organized by relevance to my emergency type on a dedicated screen,
**so that** I can browse product recommendations for the subkits I selected.

## Acceptance Criteria

1. A new screen `FillYourKitScreen` is created at route `/fill`.
2. Route guard on `/fill` implements dual-path logic: passes if `kitPath === 'essentials'` OR `selectedSubkits.length > 0`. Otherwise redirects to `/`. (Essentials users have empty `selectedSubkits` by design — the guard must check `kitPath` first. See Architecture Brief Section 7.)
3. Replace `FillKitStubModal` behavior in `OrderConfirmationScreen.tsx` with `navigate('/fill')`. Remove `FillKitStubModal.tsx`. Remove related state (`showFillKitModal`, `setShowFillKitModal`, `handleCloseFillKit`).
4. The screen reads `emergencyTypes`, `householdComposition`, and `kitPath` from the MCQ store, and `selectedSubkits` from the kit store.
5. Active category IDs determined by path: Essentials uses `ESSENTIALS_BUNDLE.map(b => b.subkit)` → `['power', 'cooking', 'medical', 'communications']`. Custom uses `selectedSubkits.map(s => s.categoryId)`.
6. Displayed categories are ordered by `getOrderedCategories(emergencyTypes[0], householdComposition)`, filtered to the active category IDs.
7. Each category is rendered as a `CategorySection` component displaying: 4px top border using `colorBase`, category icon (resolved via `iconResolver`), category name, product count badge ("{N} items" / "1 item" singular), and an expand/collapse chevron toggle.
8. The first 3 categories (or all if fewer than 3) are expanded by default. Remaining categories are collapsed.
9. Conditional category filtering: Pets category hidden entirely if `householdComposition` does not include `'pets'` (even if in selectedSubkits). Custom category excluded (no Amazon products).
10. `/fill` added to `MOBILE_EXEMPT_ROUTES` in `AppShell.tsx`. The Fill Your Kit screen is responsive and must render on mobile.
11. Responsive layout: category sections stack vertically, product card grid within each section (3-col desktop >=1024px, 2-col tablet >=640px, 1-col mobile). Grid CSS: `grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 gap-4`.
12. `CategorySection` implements expand/collapse via React state + ARIA (not `<details>`/`<summary>`):
    - `useState<boolean>(defaultExpanded)` manages open state
    - Header `<button>` has `aria-expanded`, `aria-controls`
    - Content panel has `role="region"`, `aria-labelledby`
    - Keyboard: `Enter` / `Space` toggles
    - Animation: `max-height` transition 200ms expand, 180ms collapse
    - Chevron rotates 180deg in sync
13. Page header: h1 "Fill Your Kit" + subtitle "Shop for the items to fill your emergency subkits. Each product links directly to Amazon." — focused on mount via `ref` + `tabIndex={-1}`.
14. Section header background uses `colorTint`. Product grid area inherits page background (`neutral-50`).
15. Component tests cover: correct category ordering for each emergency type, expand/collapse behavior, selected-subkits-only filtering, conditional Pets/Custom exclusion, route guard redirect (both Essentials pass and Custom pass), ARIA attributes.

### Rendering Pipeline
[Source: docs/front-end-spec-phase3-sprint3.md, Section 4.2]

```
1. Determine active category IDs (Essentials BUNDLE vs Custom selectedSubkits)
2. Get ordered categories via getOrderedCategories(emergencyTypes[0], householdComposition)
3. Filter to active categories only
4. For each category, filter products (MCQ conditions applied at screen level)
5. Skip categories with 0 products after filtering
6. First 3 → defaultExpanded={true}, rest → defaultExpanded={false}
```
