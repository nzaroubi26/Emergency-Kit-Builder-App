## Dev Notes

### Architecture Reference
[Source: docs/architecture-phase3-sprint3.md, Section 8]

All changes are to data values only. Item IDs are the invariant — all consumers reference items by `id`, not by `name`. The `KitItem` type shape is unchanged.

Key safety analysis from Winston:
- `kitStore.ts` — keys by `subkitId::itemId` → **no impact**
- `itemImages.ts` — keys by item ID → **no impact**
- `ITEM_ICON_OVERRIDES` — keys by item ID → **no impact**
- `slotCalculations.ts` / `cartCalculations.ts` — operate on typed arrays → **no impact**
- Component rendering — reads `item.name` at render time → **auto-updated**

### Removal Impact: pets-food
[Source: docs/architecture-phase3-sprint3.md, Section 8]

Users with `pets-food` in persisted `itemSelections` (localStorage) will have a dangling reference. The app handles this gracefully: the item won't render (no matching `ITEMS` entry). No crash. The entry is inert until `resetKit()` clears it.

### Icon Resolver Gap
[Source: docs/architecture-phase3-sprint3.md, Section 11]

`PawPrint` is used by `CATEGORIES.pets.icon` but is not in the icon resolver. Sprint 2 didn't surface this because `MCQHouseholdScreen` imports `PawPrint` directly (named import). But `CategorySection` in Sprint 3A will resolve icons via the resolver (same pattern as `SubkitCard`).
