# Story 2.2: Dynamic Slot State and Subkit Rendering

## Status: Done

## Story

**As a** user,
**I want** the visualizer to update in real time when I select a subkit,
**so that** I can immediately see how my selections fill the housing unit.

## Acceptance Criteria

1. When a Regular subkit is selected, its slot fills with a colored block displaying the subkit name
2. When a Large subkit is selected, it spans 2 adjacent slots with a single colored block
3. Each subkit category renders in its unique assigned color consistently throughout the app
4. Slot state updates render within 100ms of a user interaction
5. When a subkit is deselected, its slot returns to the empty + state
6. The visualizer correctly handles mixed configurations (e.g., 2 Large + 2 Regular = 6 slots)
7. Filled slots display the subkit name in legible, appropriately sized typography

## Dev Notes

### Critical Rule: Never Store Slot State
[Source: docs/architecture/11-frontend-developer-standards.md, Rule 1]
- Slot state is ALWAYS computed via `calculateSlotState()` â€” never stored in Zustand
- Components use `useSlotState()` hook which runs computation live

### Critical Rule: Dynamic Colors
[Source: docs/architecture/11-frontend-developer-standards.md, Rule 3]
- Dynamic category colors MUST use inline styles: `style={{ backgroundColor: category.colorBase }}`
- NEVER use template literal Tailwind classes â€” they get purged at build time

### Store Implementation
[Source: docs/architecture/5-state-management.md#store]
- Full implementation is provided in architecture â€” follow exactly
- Key pattern: `selectSubkit` adds with `selectionOrder`, `deselectSubkit` re-indexes remaining

### Animation Specification
[Source: docs/architecture/8-styling-guidelines.md#animation-rules]
- Slot fill: `background-color` over 220ms with `var(--ease-standard)`
- Name span: `opacity` delayed 80ms â€” these are two separate CSS transitions on two separate elements
- Large block: both rows get same background-color transition; internal divider transitions opacity to 0
- All animations GPU-composited only (transform + opacity)

### NFR2 Performance Target
[Source: docs/prd/2-requirements.md]
- Slot state changes must commit to state and begin rendering within 100ms
- Visual fill animation may complete over up to 220ms

### Testing

[Source: docs/architecture/9-testing-requirements.md]
- Store tests: reset store in `beforeEach` with `useKitStore.getState().resetKit()`
- Component tests: verify name display, color rendering, slot count
