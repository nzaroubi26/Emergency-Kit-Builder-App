## Dev Notes

### Architecture Reference
[Source: docs/architecture-phase3-sprint2.md, Section 1]

**Why `ElevationResult` struct instead of flat `Set<string>`:** The PRD specifies Q2-elevated subkits sort above Q1-only elevated. A flat `Set` loses the Q2/Q1 distinction. The `q2Elevated` subset lets `SubkitSelectionScreen` partition correctly: `q2Elevated.has(id)` = Q2 tier, `elevated.has(id) && !q2Elevated.has(id)` = Q1-only tier.

**Sort responsibility lives in SubkitSelectionScreen**, not in the elevation function. The function answers **what** is elevated. The screen decides **how** to order.

### Integration Pattern
[Source: docs/architecture-phase3-sprint2.md, Section 2c]

```typescript
// In SubkitSelectionScreen
const emergencyTypes = useMCQStore((s) => s.emergencyTypes);
const householdComposition = useMCQStore((s) => s.householdComposition);

const { elevated, q2Elevated } = computeElevatedSubkits(emergencyTypes, householdComposition);

const allCategories = Object.values(CATEGORIES);
const q2Categories = allCategories.filter((c) => q2Elevated.has(c.id));
const q1OnlyCategories = allCategories.filter((c) => elevated.has(c.id) && !q2Elevated.has(c.id));
const nonElevated = allCategories.filter((c) => !elevated.has(c.id));
const sortedCategories = [...q2Categories, ...q1OnlyCategories, ...nonElevated];
```

`Object.values(CATEGORIES)` preserves insertion order (catalog order), so filtering maintains relative order within each tier.
